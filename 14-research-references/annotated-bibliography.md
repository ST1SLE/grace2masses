# Annotated Bibliography

This file catalogues the external works that GRACE source posts explicitly reference. Each entry summarises what the cited work proves and how GRACE wires that finding into its methodology. Every factual claim below is cited back to the source post or file where it appears; the "What it proves" field reproduces only what the GRACE sources state about the cited work, not the full paper. Where the sources do not specify a piece of metadata (authors, arxiv ID, etc.), this file records that gap honestly rather than invent one.

Entries are ordered by how often and how load-bearingly GRACE leans on the cited work — the OpenAI guide and the `2601.16661` in-source documentation benchmark sit near the top because they anchor multiple GRACE rules, while the Kimi paper and the Lee reference appear lower because the sources only cite them in passing.

## OpenAI GPT-4.1 Prompting Guide
- **Authors**: not specified in sources.
- **Venue / URL**: cookbook.openai.com/examples/gpt4-1_prompting_guide#delimiters (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON").
- **What it proves**: [vendor-doc] OpenAI updated the GPT-4.1 Prompting Guide several months ago with an explicit directive not to use JSON in large contexts; XML or XML-like markup (the style Google's prompters and GRACE use) is recommended instead (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON"). [vendor-doc] The guide also cross-references Lee's "Can Long-Context Language Models Subsume Retrieval, RAG, SQL, and More?" as the research substrate for that recommendation (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON").
- **How GRACE uses it**: [author-claim] GRACE treats the OpenAI guide as the canonical vendor endorsement of XML-like markup over JSON in long contexts, alongside a parallel Google endorsement (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON"). [empirical] The author reports the same effect through his logit-analysis method, noting that the cause is JSON's "semantic noise" and the LLM's tendency to count braces across long spans, with the degradation being universal across vendors including Qwen (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON"). [author-claim] The guide is invoked as the easy answer when challenging practitioners who insist on JSON-based RAG pipelines: "follow the vendor directive rather than argue with it" (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON").
- **Source post**: (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON").

## Shunyu Yao et al. — Dyck Language in Transformers
- **Authors**: "Chinese researchers led by Shunyu Yao" (post 3948). Further author metadata not specified in sources.
- **Venue / URL**: arxiv.org/abs/2105.11115 (post 3948).
- **What it proves**: [research-backed] Yao et al. proved explicit native transformer support for Dyck Language — paired-bracket languages formalised by German mathematician Walter von Dyck — the formal characterisation of which matches the discrete-logic class TC⁰ that transformers can execute without chain-of-thought (post 3948). [research-backed] The class of formats compatible with Dyck Language — and therefore natively parseable by GPT — includes XML, HTML, JSON, and programming languages based on paired brackets (post 3948). [research-backed] YAML is not Dyck-compatible, which the post cites as the reason Google banned YAML libraries in Gemini's Code Execution sandbox to prevent the model from degrading while trying to handle it (post 3948).
- **How GRACE uses it**: [author-claim] GRACE uses the Dyck-Language result to justify its consistent preference for XML-like markup and to explain why the author adds paired `START`/`END` tags inside Python comments — the comments themselves are not Dyck-friendly alone, but the paired tags make the markup parseable in a single pass without degradation (post 3948). [author-claim] The same result underpins the broader recommendation to convert YAML configurations into Dyck-compatible formats and the observation that YAML is often inserted into AI agents "by incompetence" by second-role vendor engineers (post 3948).
- **Source post**: (post 3948).

## In-source Documentation Benchmarks
- **Authors**: not specified in sources.
- **Venue / URL**: arxiv.org/abs/2601.16661v1 (post 3889).
- **What it proves**: [research-backed] The work demonstrates four causally important effects of in-source documentation on agent performance (post 3889):
  1. PURPOSE (goal and intent) is critically important: an agent can manage perfectly well without a DESCRIPTION of a function, but removing the "why" causes proven degradation.
  2. Shorter comments are more effective; long descriptions are either neutral or cause degradation.
  3. English is the best language for code comments; other languages ("sovereign-AI"-style non-English commentary) cause agent degradation.
  4. Descriptions must be adjacent to the code — if documentation is separated from code by a file read or an MCP hop so that tokens of distance accumulate, the agent degrades.
- [research-backed] Effect sizes are large: up to -95% degradation when structure is broken, up to +435% improvement when embedded descriptions are well done (post 3889). [research-backed] Coverage: results are tested on a large set of programming languages and across LLMs from different vendors, which is why the post calls out the paper's reliability (post 3889). [research-backed] The same work implies that stale documentation is toxic: if docs lag behind code, agents degrade on false information, and LLMs know this from their training distribution, which is consistent with the observed high refusal rate to read separate documentation (post 3889).
- **How GRACE uses it**: [author-claim] GRACE elevates PURPOSE to a first-class, load-bearing contract field rather than treating it as optional narrative; the benchmark is cited as independent proof that an agent can skip DESCRIPTION but not PURPOSE (post 3889). [author-claim] GRACE applies aggressive minimisation to all embedded descriptions, favouring short, keyword-dense forms over full sentences, and verifies each compression with logit-level checks (post 3889, post 3892). [author-claim] GRACE forbids "sovereign-AI" non-English comments in code that will be read by LLMs, citing the benchmark's proof that only English avoids degradation (post 3889). [author-claim] GRACE insists that embedded documentation sit immediately next to the declaration it describes (before the `def` line in Python), because any token-level separation caused by reading a different file or hopping through an MCP server is proven to degrade agents (post 3889).
- **Source post**: (post 3889).

## ShortenDoc
- **Authors**: not specified in sources.
- **Venue / URL**: dl.acm.org/doi/10.1145/3735636 (post 3892).
- **What it proves**: [research-backed] ShortenDoc experiments operate on Python docstrings and prove that full prepositional phrases inside a docstring degrade LLM code generation. In contrast, aggressively pruning the text to leave only key concepts — "mutilating" the prose — raises HumanEval pass rates by approximately 10% and allows compression of up to 40% of the docstring's tokens either without quality loss or with a small improvement (post 3892). [research-backed] The authors additionally validate the effect via perplexity closeness: they check that the model predicts tokens correctly from the compressed description, which proves the model still understands it (post 3892). [research-backed] The paper also proposes a small companion neural network named "ShortenDoc" to generate such compressed descriptions, effectively productising wenyan-style compression for code (post 3892). [research-backed] The paper further nuances the earlier degradation result: docstrings are broken for GPT mainly once sparse attention engages — on small contexts where sparse attention does not kick in, a standard docstring can still work as an SFT-style hint, and the sliding window covers roughly the last 1000 tokens during generation, so a docstring below a declaration still contributes (post 3892).
- **How GRACE uses it**: [author-claim] GRACE uses the ShortenDoc result to justify its wenyan-prompting style — KEYWORDS, LINKS, and similar compressed sections — as an evidence-based compression technique rather than a stylistic quirk (post 3892). [empirical] The author applies his own two-step validation before accepting a compression into GRACE templates: (1) logit-level check that the compressed phrase predicts the same downstream tokens, and (2) a blind test where GPT is asked what the compressed phrase means without being shown code or instructions; only compressions that pass both checks ship (post 3892). [author-claim] The phrase "GPT has already invented a secret transformer language; you just don't speak it yet" comes from this section of the argument, framing compressed semantics as the model's native dialect (post 3892).
- **Source post**: (post 3892).

## On the Impact of Code Comments for Automated Bug-Fixing
- **Authors**: not specified in sources.
- **Venue / URL**: researchgate.net/publication/400340807 (post 3890).
- **What it proves**: [research-backed] The paper demonstrates that "Captain Obvious" comments — ones that restate what the code already says — are not needed by an LLM because it reads code fluently; worse, they occupy token budget without adding semantic value (post 3890). [research-backed] However, when embedded documentation exposes architectural intent rather than surface behaviour, bug-fix accuracy improves by up to 3x. Attention analysis in the paper shows that the AI agent specifically attends to the intent tokens when reconstructing the logic of a broken method (post 3890).
- **How GRACE uses it**: [empirical] The author reports he had discovered the same effect empirically before the paper, and the paper is cited as the independent confirmation (post 3890). [author-claim] GRACE therefore devotes explicit contract machinery to architectural intent: references to architectural patterns, and dedicated RATIONALE / AAG sections inside module contracts, both of which are positioned specifically to capture the "why" tokens the attention analysis showed LLMs rely on (post 3890). [author-claim] The same result is cited as the reason GRACE discourages pure-description comments in favour of intent-carrying ones.
- **Source post**: (post 3890).

## Google CodeT5+ / JML Contract Recovery
- **Authors**: not specified in sources.
- **Venue / URL**: dl.acm.org/doi/10.1145/3689484.3690738 (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy").
- **What it proves**: [research-backed] The researchers used the Java Modeling Language (JML) — a formal specification language for annotating Java code in the Design-by-Contract paradigm — as the target form of contract (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy"). [research-backed] Using a fine-tuned Google CodeT5+ network, they were able to reconstruct contracts for existing code at 90-97% quality depending on the class of test, with the main focus being the recovery of critical input/output conditions and invariants (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy").
- **How GRACE uses it**: [author-claim] GRACE cites this result as the scientific backing for adding contracts to legacy codebases via an LLM pipeline, arguing that even the relatively primitive JML-style contracts recovered in the paper are enough to almost eliminate the risk of the AI "destroying" code during edits (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy"). [empirical] The author reports positive personal experience reconstructing contracts on legacy code, with quality evaluated by an LLM reviewer rather than formal tests, and frames the combination of paper + practice as evidence that the industry is close to "total adoption of contract programming" (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy"). [author-claim] GRACE's own AI-contracts, however, are explicitly not classical DbC — they are closer to Python docstrings and Spec-Driven Development micro-specs; the CodeT5+/JML paper is used for the legacy-recovery argument rather than as a model for GRACE's contract form (contracts_as_anchor.md §"Почему AI-contracts не равны классическому DbC").
- **Source post**: (contracts_as_anchor.md §"Научная опора на восстановление контрактов в legacy").

## Kimi paper on MCP Training Distribution
- **Authors**: not specified in sources.
- **Venue / URL**: not specified in sources; cited only as "the Kimi paper" (post 3834).
- **What it proves**: [research-backed] According to the Kimi paper as summarised in post 3834, the top-1000 MCP servers are inside the training distribution of GPT-class models, which means tools from market leaders — the post names Asana and JIRA specifically — have a high probability of being usable by agents inside long trajectories (post 3834). [research-backed] By implication, anything outside that top-1000 cohort — custom, DIY, "kulibin"-style MCPs — was not seen during training and therefore has a much lower probability of being invoked reliably by an agent (post 3834).
- **How GRACE uses it**: [author-claim] GRACE cites the Kimi paper as the scientific foundation for its strong preference for standard industry tooling over custom tooling in the agentic stack (post 3834). [empirical] The author's own experience matches the paper: he built a "kulibin CLI" for tests alongside standard pytest, and the LLM either ignored his CLI or hallucinated around it, while pytest worked "like a rock", so he removed the custom CLI and returned strictly to the industrial standards the LLM was trained on (post 3834). [author-claim] The same argument is used to explain why custom MCPs can be silently refused by agents — unlike CLI commands where the agent is at least forced through bash — and why partners shipping custom MCPs compete against the vendor's $10-billion training cluster and lose (post 3821, post 3834).
- **Source post**: (post 3834; with related argument in post 3821).

## Notes on citation fidelity

- [author-claim] This bibliography deliberately records "not specified in sources" for metadata (paper authors, arxiv IDs, further benchmarks) that the GRACE source posts do not include. The PLAN.md rule against invention (PLAN.md §4) is treated as strict, even where the assistant could plausibly guess a value (PLAN.md §4).
- [author-claim] Where the source cites a result via another paper (the Lee reference) or via a summary tag ("per the Kimi paper"), this bibliography preserves that indirect citation rather than elevating it to a direct one. This keeps the evidence chain honest: the sources reach these papers transitively, and so should downstream readers (xml_mapping_anchors.md §"Почему автор предпочитает XML-like разметку вместо JSON"; post 3834).
- [author-claim] The "How GRACE uses it" sections distinguish vendor-doc and research-backed findings from author-claim and empirical extensions. Where the author reports having discovered an effect empirically before the paper was published (post 3890), both the research-backed and empirical labels appear together.
